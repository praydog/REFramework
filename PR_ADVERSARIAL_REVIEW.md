# PR Adversarial Review

## Summary

Eight findings total: 1 critical, 3 high, 3 medium, 1 low. The most severe is an init-order defect that silently disables the AddVectoredExceptionHandler filter for the six games that depended on it. Two further findings are wrong-struct-layout reads on MHWILDS/RE9 (REType canonical layout) and RE7 (RETransform joint array), both of which are actively exercised by shipped mod code. The remaining findings are latent landmines in the renderer SDK that will fire the moment a plugin or future mod wires up `TargetState::get_rtv*`, `ConstantBufferDX12::get_desc()`, or `Overlay::get_b8g8r8a8_unorm_*`.

## Findings

### Critical — AddVectoredExceptionHandler filter silently disabled for every game in universal build

- **Location**: `src/mods/IntegrityCheckBypass.cpp:2243-2254`, call chain from `src/Main.cpp:141`
- **Mechanism**: `hook_add_vectored_exception_handler()` runs synchronously from `DllMain(DLL_PROCESS_ATTACH)` at `src/Main.cpp:141`. Its first line is `if (sdk::GameIdentity::get().tdb_ver() < 73) return;`. But `sdk::GameIdentity::initialize()` is *not* called synchronously — it is called inside `startup_thread()` at `src/Main.cpp:96`, which is dispatched via `CreateThread` at `src/Main.cpp:144` and therefore runs after DllMain returns. At the time of the VEH check the singleton still holds the default-initialised `m_tdb_ver == 0`, so `0 < 73` is true and the hook is never installed. The origin/master code used `#if TDB_VER >= 73` which was evaluated at compile time per per-game target, so the hook installed correctly. In universal, the behaviour inverts: the hook is never installed on *any* game.
- **Reachability**: Every universal process load. Runs once at DLL attach; failure is permanent for the process lifetime. The game's own anti-debug `AddVectoredExceptionHandler` calls that the filter was designed to reject are now passed through, so any detection routine wired to a rogue VEH fires unconditionally.
- **Games affected**: `DD2` (tdb 73), `STARFORCE` (tdb 78), `MHWILDS` (tdb 81), `MHSTORIES3` (tdb 82), `RE9` (tdb 83), `PRAGMATA` (tdb 83). These are the games whose per-game master build had `TDB_VER >= 73` and therefore installed the filter. Per `shared/sdk/GameIdentity.cpp:230-288`.
- **Reproduction**: Launch any of the six affected games with the universal `dinput8.dll`. Inspect the log — `"[IntegrityCheckBypass]: Hooking AddVectoredExceptionHandler..."` never appears. Compare against an origin/master per-game DLL (e.g. the RE9 target), which prints that line.
- **Suggested fix**: Move `GameIdentity::initialize()` up into `DllMain` (before the `IntegrityCheckBypass::*` calls) so the three DllMain-time integrity hooks see a valid singleton. Exe-name detection is a single `GetModuleFileNameW` — safe to run synchronously under loader lock.

### High — Universal REType canonical layout wrong for MHWILDS and RE9; `retype_is_large_layout()` predicate wrong for MHSTORIES3 and PRAGMATA

- **Location**: `shared/sdk/ReClass.hpp:9-12`, `shared/sdk/RETypeLayouts.hpp:47-50`; concrete struct mismatch between `shared/sdk/ReClass_Internal_RE8.hpp:125-147` (canonical, super @ 0x38, size 0x60) and `shared/sdk/ReClass_Internal_MHWILDS.hpp:125-147` / `shared/sdk/ReClass_Internal_RE9.hpp:125-147` (super @ 0x40, size 0x68)
- **Mechanism**: `ReClass.hpp` unconditionally includes `ReClass_Internal_RE8.hpp` in universal builds, fixing `REType::super` at 0x38 and `REType::classInfo` at 0x58. Per the MHWILDS/RE9 ReClass dumps, the real runtime `REType` is 0x68 bytes with super at 0x40, childType at 0x48, chainType at 0x50, fields at 0x58, classInfo at 0x60. Every direct field access (`t->super`, `t->classInfo`, `t->childType`, `t->fields`, `t->chainType`) compiled into the universal DLL reads at the RE8 offset, so on MHWILDS/RE9 `t->classInfo` returns what the real game stores as `fields` (a `REFieldList*`), `t->super` returns 8 bytes of padding, and so on. `RETypeLayouts.hpp` was added to paper over this with `reinterpret_cast<reclass_mhwilds::REType*>` accessors, but (a) only `get_typeCRC` is wired in (single call site at `RETypeDefinition.cpp:968` — and typeCRC happens to sit at 0x2C in both layouts, so that accessor is a no-op); every other access site uses the raw `t->...` form. (b) The predicate `retype_is_large_layout()` at `RETypeLayouts.hpp:47` gates on `tdb_ver() >= 81`, which incorrectly includes MHSTORIES3 (tdb 82, real REType 0x60, super @ 0x38) and PRAGMATA (tdb 83, real REType 0x60, super @ 0x38). If/when the accessor is wired up, it will dispatch MHSTORIES3 and PRAGMATA to the 0x68 layout and read 8 bytes past the real `super`/`childType`/etc. The predicate should be `is_mhwilds() || is_re9()`, not a TDB-version gate.
- **Reachability**: Every managed-object inspection path. Direct `t->classInfo` is accessed on hot paths: `shared/sdk/REComponent.hpp:66-70`, `shared/sdk/REManagedObject.cpp:251,283,324,364,370,400,417,440,488,497,502,504,511`, `shared/sdk/Renderer.cpp:205,223,263,339,1137`, `src/mods/IntegrityCheckBypass.cpp:290`. `t->super` is accessed on the dev UI path at `src/mods/tools/ObjectExplorer.cpp:1938-1939,2123,2129`. The affected code runs every frame the RE-Engine metadata system is used on MHWILDS/RE9.
- **Games affected**: `MHWILDS` (tdb 81), `RE9` (tdb 83) — direct field access reads wrong offset. `MHSTORIES3` (tdb 82), `PRAGMATA` (tdb 83) — accessor dispatches to wrong layout (latent, fires only when `re_type_accessor::get_super`/`get_childType`/etc. gets wired in; today only `get_typeCRC` is used and that field happens to share the same offset in both layouts).
- **Reproduction**: On MHWILDS, invoke any managed-object class info lookup (ObjectExplorer opens the type tree, or any call into `utility::re_managed_object::get_field`). `t->classInfo` silently returns the game's `REFieldList*` cast to `REClassInfo*`. Downstream `classInfo->parentInfo` dereferences will either crash or return wrong type metadata. The author's PLAN marks MHWILDS as "Full" compatibility (line 96 of PLAN_V2_MONOLITHIC.md), which contradicts the ReClass dump in `ReClass_Internal_MHWILDS.hpp` that says super sits at 0x40.
- **Suggested fix**: Either (a) teach `ReClass.hpp` to conditionally include the 0x68 layout when `is_mhwilds() || is_re9()` and route every direct field access through `utility::re_type_accessor::*`, or (b) change `retype_is_large_layout()` to `gi.is_mhwilds() || gi.is_re9()` AND convert every `t->super|classInfo|childType|chainType|fields` call site in the tree to the accessor. Option (b) requires a grep-everything pass or MHWILDS/RE9 will still break at the un-converted sites.

### High — `get_joint_array_data()` missing RE7 dispatch; RE7 joint arrays read at the wrong offset in universal

- **Location**: `shared/sdk/RETransform.hpp:66-82`
- **Mechanism**: Modern RE7 (TDB 70) shares its RETransform layout with RE2/RE3 TDB 70 — joints at offset 0xD0, not 0xD8. Evidence: the legacy per-game build in `shared/sdk/ReClass.hpp:49-54` explicitly includes `ReClass_Internal_RE2_TDB70.hpp` for modern RE7 (`#elif defined(RE7) … #ifdef RE7_TDB49 … #else #include "ReClass_Internal_RE2_TDB70.hpp" #endif`), and that ReClass dump places `REJointArray joints` at 0xD0 (`ReClass_Internal_RE2_TDB70.hpp:300`). The recent fix commit `c9d8e32b` added `get_joint_array_data()` with a runtime dispatch that hardcodes `this + 0xD0` only when `gi.is_re2() || gi.is_re3()`. RE7 is not in the branch, so for RE7 the call falls through to `transform.joints.data`, which is compiled against the RE8 canonical ReClass layout (joints at 0xD8). Every RE7 joint lookup therefore reads 8 bytes past the real `joints.data` pointer (into `joints.matrices` or `joints.numElements`, depending on REJointArray layout), returning garbage. The analogous `get_joint_matrices()` at line 66-72 has the same missing RE7 case (hardcoded 0xC0 only for RE2/RE3, RE7 falls through to the RE8 `transform.joints.matrices`).
- **Reachability**: Active. Called from `utility::re_transform::get_joint(transform, index)` at `RETransform.cpp:189-216` and `get_joint(transform, std::wstring_view name)` at `RETransform.hpp:85-118`, plus `get_all_children` at `RETransform.hpp:206`, `get_immediate_children` at line 265. Consumers include `src/mods/FirstPerson.cpp` (head/face bone zeroing), VR head-culling, `src/mods/tools/ChainViewer.cpp`, etc. The VR FirstPerson head-hide feature is precisely the scenario the c9d8e32b commit message cites as broken on RE2 — the same bug is left in place for modern RE7.
- **Games affected**: `RE7` (tdb 70, modern). Legacy RE7_TDB49 is out of the universal roster per PLAN line 103.
- **Reproduction**: Run the universal DLL on RE7 (modern), enable VR + FirstPerson head-hide, observe that the head bone is never zeroed (joint lookup returns garbage → name compare fails → bone not found). Or set a breakpoint in `get_joint_array_data` with `gi.is_re7()` true and observe the `this + 0xD8` path is taken.
- **Suggested fix**: Change both `get_joint_matrices` and `get_joint_array_data` to `if (gi.is_re2() || gi.is_re3() || gi.is_re7())`. Same RE2_TDB70 offsets apply.

### High — `TargetState::m_desc` at wrong compile-time offset for 11 of 14 games

- **Location**: `shared/sdk/Renderer.hpp:185-264` (TargetState class + universal accessors), `shared/sdk/renderer/RenderResource.hpp:28-41` (`get_runtime_size`)
- **Mechanism**: `RenderResource` contains two conditional padding pointers — `_tdb73_padding` (gated `SF6 || TDB_VER >= 73 || TDB_VER <= 67 || RE4`) and `tdb82_padding` (gated `TDB_VER >= 82`). In universal, TDB_VER=84 forces both into the compiled struct, so `sizeof(RenderResource) == 0x20`. But `RenderResource::get_runtime_size()` correctly reports the real size per game: 0x10 for RE7/RE2/RE3/RE8/MHRISE, 0x18 for RE4/SF6/DD2/MHWILDS/STARFORCE/DMC5, 0x20 for MHSTORIES3/RE9/PRAGMATA. `TargetState : RenderResource` has its `Desc m_desc` member at the compile-time offset `sizeof(RenderResource) == 0x20`, not the runtime size. Therefore on every game whose real RenderResource is smaller than 0x20 — 11 of 14 — the compiled `&m_desc` points 8–16 bytes past the real `m_desc`. The universal `get_rtv_count()` / `get_rtvs_ptr()` formulas (Renderer.hpp:202-210) add only a `pad = (tdb_ver() <= 67 ? 8 : 0)` correction for DMC5's missing `_unk_pad` field, which is unrelated to the RenderResource size mismatch and fails to close the 8/16-byte gap. Net result: `get_rtv_count`, `get_rtv`, `set_rtv`, `get_desc`, `clone()` all dereference wrong memory on 11 games.
- **Reachability**: Latent in current tree — no call site in `src/`, `shared/sdk/`, or `csharp-api/` exercises `TargetState::get_native_resource_d3d12()` or the `clone()` methods. BUT the surface is exposed: `RenderLayer::get_target_state_resource_d3d12(name)` at `Renderer.hpp:336`, and the specialisations `Output::get_output_target_d3d12`, `Scene::get_motion_vectors_d3d12`, `Scene::get_post_main_target_d3d12`, `Scene::get_hdr_target_d3d12`, `Scene::get_g_buffer_target_d3d12`, `Scene::get_hdr_target`, `Scene::get_depth_target` all funnel into `state->get_native_resource_d3d12()` which then calls the broken `get_rtv(0)`. Any plugin, Lua script, or future mod that reaches for a named render-target via the reframework plugin ABI will hit the broken path. The PLAN document already flags `ConstantBufferDX12::get_desc()` as a latent variant of the same class of bug (and `sizeof(ConstantBuffer)` being 0x20-inflated); `TargetState::m_desc` is a peer of that bug but is *not* documented in the safety-notes commit `daee901f`.
- **Games affected**: `DMC5` (tdb 67), `RE7` (tdb 70), `RE8` (tdb 69), `RE2` (tdb 70), `RE3` (tdb 70), `MHRISE` (tdb 71), `RE4` (tdb 71), `SF6` (tdb 71), `DD2` (tdb 73), `STARFORCE` (tdb 78), `MHWILDS` (tdb 81). `MHSTORIES3`, `RE9`, `PRAGMATA` happen to be correct (real RR = 0x20).
- **Reproduction**: Add a test plugin that calls `reframework::API::sdk::renderer::get_scene_layer()->get_hdr_target_d3d12()` (or the equivalent through `get_post_main_target_d3d12`), invoke it on e.g. DD2. The returned `ID3D12Resource*` will be bogus. In-memory: dump `target_state + 0x20` on DD2 and compare against the real game's RenderTargetState — real `m_desc` lives at `target_state + 0x18`.
- **Suggested fix**: Replace `&m_desc` in `get_rtv_count`/`get_rtvs_ptr`/`get_desc`/`clone` with `(uintptr_t)this + RenderResource::get_runtime_size()` and drop the `pad` kludge (DMC5 is now handled by `get_runtime_size()` returning 0x18 + the Desc-level `_unk_pad` offset which should be encoded in the Desc struct, not layered on top of m_desc). Audit every `x->m_desc` direct access; any that exist outside Renderer.hpp's own accessors are also broken.

### Medium — `ChainViewer` renders with a single hardcoded regenny layout in universal; wrong pointer deref on DMC5 / MHRise-TDB70 / RE4 / SF6

- **Location**: `src/mods/tools/ChainViewer.cpp:18-24` (universal includes), usage in `src/mods/tools/ChainViewer.cpp:264-500`
- **Mechanism**: In universal, ChainViewer unconditionally includes `sdk/regenny/re2_tdb70/via/motion/Chain.hpp` + `ChainCollisionTop.hpp` + `ChainCollisions.hpp` and compiles its entire `regenny::via::motion::Chain*` access graph against that one layout. The legacy per-game build chose between five different regenny layouts (re3, re8, mhrise, mhrise_tdb71, re4, sf6, re2_tdb70) based on `TDB_VER` and game macro. The author's own comment at line 19-21 says: *"Games with TDB < 69 or MHRISE TDB==70 use flat collision layouts with different struct sizes, so the chain viewer may not display correctly for those games."* The failure mode is worse than "may not display" — the loop at lines 313-319 does `chain->CollisionData.collisions[i].num_collisions` where `CollisionData` is at an unchecked offset, then dereferences `collider.joint`, `collider.pair_joint`, `collider.sphere.pos`, `collider.capsule.p0`, `collider.offset` as pointers/Vector4f at offsets that only match the RE2_TDB70 layout. On DMC5/MHRISE-TDB70/RE4/SF6 these offsets read game-internal state of the wrong type — `collider.joint` can return a non-null pointer to something that is not an `::REJoint`, which is then passed to `sdk::get_joint_position` / `sdk::get_joint_name`, dereferencing uncontrolled memory.
- **Reachability**: Dev-UI feature only, user-toggleable (`m_enabled->value()` gate). User must open the REFramework menu and enable the Chain Viewer. Once enabled, it runs per-frame. Not active by default.
- **Games affected**: `DMC5` (tdb 67), `RE4` (tdb 71), `SF6` (tdb 71), `MHRISE` (tdb 71) — and potentially `DD2` (tdb 73), `MHWILDS` (tdb 81), `STARFORCE` (tdb 78), `MHSTORIES3` (tdb 82), `RE9` (tdb 83), `PRAGMATA` (tdb 83) depending on how divergent their real Chain layout is from re2_tdb70. At minimum, the four legacy builds that used dedicated regenny files (re3, re8, mhrise, mhrise_tdb71, re4, sf6) are confirmed to have distinct struct layouts.
- **Reproduction**: Launch DMC5 with universal DLL, open REFramework ⇒ Tools ⇒ Chain Viewer and enable it. Any chain-component-owning actor on screen will hit the render loop. Likely crash or garbled collider visuals.
- **Suggested fix**: Either gate the whole ChainViewer behind `is_re2() || is_re3() || is_re8()` for now, or split the rendering loop into per-layout branches using the existing ViaDispatch-style namespaced-header pattern.

### Medium — `RETypeDB_DISPATCH` routes TDB 78 (STARFORCE) to `tdb74` layout but the author has not verified STARFORCE's real TDB struct size

- **Location**: `shared/sdk/RETypeDB.hpp:1881`, `:1898`, `:1936`, `:1952`, `shared/sdk/RETypeDefDispatch.hpp:158`
- **Mechanism**: Every dispatch macro — `TDB_DISPATCH`, `TDB_DISPATCH_69`, `TDB_DISPATCH_PTR`, `TDB_DISPATCH_PTR_69`, `TIMPL_DISPATCH` — has an explicit `case 78: → tdb74::TDB` path commented `/* STARFORCE: same layout as TDB74 */`. There is no `tdb78` struct definition, no ReClass dump for STARFORCE (the repository copies MHSTORIES3 in lieu — `ReClass.hpp:26: #elif defined(STARFORCE)  #include "ReClass_Internal_MHSTORIES3.hpp" // Copy of MHSTORIES3 (for now)`), and no runtime validation of the `numTypes`/`numMethods`/etc. header fields. If STARFORCE diverges from TDB 74 anywhere — which is plausible, as TDB 74→78 spans several engine releases — every TDB read (`get_num_types`, `get_modules_ptr`, `get_params_ptr`, etc.) returns data from the wrong offset. The dispatch-macro default branch for `tdb_ver() >= 84` never covers STARFORCE because 78 is captured explicitly.
- **Reachability**: Every TDB header access. `get_types_ptr` / `get_methods_ptr` / etc. are called on every `find_type_definition` / `find_native_method` / `invoke_object_func` — i.e., per-frame by every mod that calls into the managed type system.
- **Games affected**: `STARFORCE` (tdb 78, per `GameIdentity.cpp:260-268`).
- **Reproduction**: On STARFORCE, check the `[GameIdentity] Detected game: STARFORCE (TDB ver 78 …)` log line and then count of types logged by the SDK bootstrap. If `get_num_types()` returns nonsense, the REFramework type index is corrupt.
- **Suggested fix**: Either (a) add a static `static_assert`-equivalent runtime check on bootstrap that confirms the first few TDB fields look sane (e.g., magic/version match, numTypes plausible) and bails gracefully on STARFORCE if the layout is wrong, or (b) produce a dedicated `tdb78` struct and ReClass dump before shipping STARFORCE support. The current `// Copy of MHSTORIES3 (for now)` comment in `ReClass.hpp:26` is itself an admission that STARFORCE is untested.

### Medium — `Overlay::s_b8g8r8a8_unorm_*_offset` constants used for every game despite being RE4/SF6-only

- **Location**: `shared/sdk/Renderer.hpp:620-636`
- **Mechanism**: Four constants (`s_b8g8r8a8_unorm_textures_offset=0x288`, `s_r8g8b8a8_unorm_textures_offset=0x260`, `s_b8g8r8a8_unorm_target_state_offset=0x1A0`, `s_b8g8r8a8_unorm_target_only_state_offset=0x1E0`) are declared as compile-time constexpr. They are verified only for RE4/SF6 per the TODO comment above them. In the universal build they apply to every game; `Overlay::get_b8g8r8a8_unorm_textures()` / `get_r8g8b8a8_unorm_textures()` / `get_b8g8r8a8_unorm_target_states()` / `get_b8g8r8a8_unorm_target_only_states()` add these offsets to an Overlay pointer unconditionally. This was already flagged by the author in commit `daee901f`'s safety notes, but the code is not gated — it just has a comment.
- **Reachability**: Plugin-exposed helpers. Not called from `src/` today. Any plugin that reaches for these via the reframework plugin ABI on a non-RE4/SF6 game will read from the wrong offset.
- **Games affected**: All 14 games except `RE4` and `SF6`.
- **Reproduction**: A plugin calls e.g. `overlay->get_b8g8r8a8_unorm_textures()` on DMC5 — receives a `Texture**` pointing into unrelated Overlay fields.
- **Suggested fix**: Replace the four constants with a `get_s_b8g8r8a8_*_offset()` runtime dispatcher keyed on `is_re4() || is_sf6()` (return `std::optional<size_t>` and return `nullopt` for other games so callers can refuse rather than silently read garbage).

### Low — `ChainViewer::on_initialize` never checks whether the active game's Chain layout is compatible

- **Location**: `src/mods/tools/ChainViewer.cpp:69-72`
- **Mechanism**: `on_initialize` returns `std::nullopt` unconditionally. Unlike other game-specific mods, there is no early-out for games on which the hardcoded re2_tdb70 Chain layout is known wrong. Combined with the Medium finding above, this means the mod *advertises support* for all 14 games but only renders correctly on a subset. Not a runtime defect per se, but a UI-surface inconsistency: the toggle is visible on every game, even ones where enabling it is known by the author to misbehave.
- **Reachability**: UI only — no effect unless the user enables the viewer.
- **Games affected**: Cosmetic, all games with divergent Chain layouts.
- **Reproduction**: Open the mod list on DMC5; ChainViewer is offered as an enabled-capable option.
- **Suggested fix**: `on_initialize` returns `std::nullopt` only for `is_re2() || is_re3() || is_re8()` and returns a "not supported" string otherwise, or gate the enable toggle in the UI.

## Out-of-scope observations

- `shared/sdk/Application.cpp:56` declares a static local of a captured lambda inside `get_functions()` (`static auto functions_offset = [this]() -> std::optional<int32_t> { … }`). The lambda captures `this` so the static is initialised on first call, which is fine given the function is called after DllMain — but if a future refactor moves any pattern-scan-based Application lookup into the DllMain critical path, the static will be initialised under the loader lock with a broken GameIdentity. Worth a comment stating the invariant.
- The `copy_texture` lambda pair in `Renderer.cpp:795-919` uses two `static auto func = []() { … }()` initializers, one in each of the legacy/modern branches. The universal if-chain at lines 909-919 picks one branch at runtime. If `tdb_ver()` were ever to flip mid-process (it does not today, but `GameIdentity` deliberately leaves a door open for that via "the TDB binary header auto-detect" pathway described in PLAN §2), only the first-visited branch's `func` static would ever initialise; the other would stay null. Document the immutability of `tdb_ver()` post-initialize, or guard against flips.
- `shared/sdk/RETypeDB.hpp:1878` maps `case 71: case 72:` to `tdb71::TDB`. There is no game currently mapped to TDB 72 in `GameIdentity::derive_engine_params()`, so case 72 is dead — remove or leave an explicit comment. Not a defect, but the spec surface is larger than any live code exercises.
- `shared/sdk/ViaDispatch.hpp:94-107` routes `STARFORCE`, `MHWILDS`, `RE9`, `PRAGMATA` to the `re9` regenny namespace by default. STARFORCE has not been confirmed to share RE9's Window/SceneView layout — given that the recent commit `842fabf3` had to specifically pull MHSTORIES3 out of that default and redirect it to the DD2 layout ("MH Stories 3 blurriness — Window uses DD2 layout"), there is a plausible risk that STARFORCE needs its own redirect too. No repro data available in this review's budget.
- `hook_lightshaft_draw` in `src/mods/Hooks.cpp:553-591` is entirely inside `#if 0`, yet the `HOOK_LAMBDA(hook_lightshaft_draw)` at `Hooks.hpp:149` is still added to the hook list under `REFRAMEWORK_UNIVERSAL`. It returns `std::nullopt` unconditionally so it's harmless — but the dead-code form reads like an oversight; either delete the `#if 0` body or drop the hook registration.

## Coverage gaps

- **STARFORCE and PRAGMATA runtime validation**: I do not have access to running instances of either. Every STARFORCE/PRAGMATA-specific finding is derived from (a) the ReClass dumps that ship in the repo, (b) the TDB version tables in `GameIdentity.cpp`, and (c) diff against the reference `re2-barebones` repository. Offsets I could not verify: the real STARFORCE TDB header layout (dispatch routes it to tdb74 — plausible but unverified); STARFORCE's `via::Window`/`via::SceneView` (routed to re9 namespace by default in ViaDispatch); STARFORCE's REType size (included by copy of MHSTORIES3's ReClass, but MHSTORIES3 is 0x60 — might be right, might not).
- **Modern RE7 TDB70**: I inferred RE7's RETransform layout from the per-game-build-include choice in `ReClass.hpp:49-54` (which selects RE2_TDB70 ReClass for modern RE7). I did not run RE7 to confirm `joints` is at 0xD0 — the claim is founded on the assumption that the author's per-game build was correct.
- **DX11 code paths**: I focused on DX12 paths (get_target_state_resource_d3d12, get_native_resource_d3d12) because they are wired into VR/BackBufferRenderer. The DX11 analogues may have parallel issues I did not inspect.
- **VR module (`src/mods/VR.*` and `src/mods/vr/**`)**: These are the heaviest consumer of the renderer SDK and the most likely to hit the TargetState and RE7 RETransform bugs in production. A dedicated pass through the VR module would likely surface additional reachability facts upgrading finding #4 from Medium (latent) to High (active). I did not do that pass within my time budget.
- **Legacy TDB-version runtime re-detection**: PLAN §2 describes a mechanism where `GameIdentity::tdb_ver()` is supposed to be updated after the binary TDB header is read, so e.g. an RE7 running a TDB49 binary would auto-downgrade. `GameIdentity.cpp:136-147` explicitly disables that: *"This function does not perform runtime override/mutation of m_tdb_ver for legacy binaries"*. I treated this as a documented non-goal, not a defect — but any code path that assumes the re-detection happens will silently misbehave on legacy binaries. The `GameIdentity` doc comment in `GameIdentity.hpp:22` still promises it ("may be updated after TDB header is read"), which contradicts the implementation.
- **IntegrityCheckBypass body (post-VEH hook)**: I traced the DllMain-time entrypoints but did not audit the full SHA3/RSA bypass logic for further DllMain-time GameIdentity dependencies. The VEH-hook bug is the one I proved; there may be additional early-init paths that depend on the singleton that I did not chase down.
